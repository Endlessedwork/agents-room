# Phase 15: Presets CRUD - Research

**Researched:** 2026-03-21
**Domain:** SQLite schema migration, Next.js App Router API routes, Zustand store patterns, React form UX
**Confidence:** HIGH

## Summary

Phase 15 introduces a persistent `presets` table in SQLite to replace the static `AgentPresets.ts` array. Three system presets (Devil's Advocate, Code Reviewer, Researcher) are seeded via a Drizzle migration. Users can create new presets, save an existing agent's config as a preset, edit user presets, and delete user presets. System presets are immutable (protected by an `isSystem` flag).

The existing codebase already has all the supporting infrastructure: Drizzle ORM with migration workflow, Zod validation, nanoid for IDs, Zustand stores for client state, Dialog components for confirmations, and a fully-formed AgentForm component that supports preset-seeding via props. The primary work is additive: new DB table, new API routes, a new Zustand `presetStore`, and new/modified UI pages.

The static `AGENT_PRESETS` array in `AgentPresets.ts` is currently consumed by the agents page (to render "Preset Templates") and by the new-agent page (to resolve `?preset=` query param). Both usages must migrate to fetching from the API. The static file can be kept temporarily for the seed data and then gutted — or left as the seed source and simply no longer used for runtime rendering.

**Primary recommendation:** New `presets` DB table seeded with the 3 system presets; API routes at `/api/presets` (GET/POST) and `/api/presets/[id]` (PUT/DELETE); new `usePresetStore` Zustand store; presets management UI at `/presets`; "Save as preset" button added to AgentForm.

### Pattern 1: DB Schema — presets table

The `presets` table mirrors the agent config fields exactly, plus `is_system` and `isSystem` flags to prevent deletion of seeded records.

```typescript
// Source: existing schema.ts pattern
export const presets = sqliteTable('presets', {
  id: text('id').primaryKey(), // nanoid for user; stable slug for system (e.g. 'devils-advocate')
  name: text('name').notNull(),
  avatarColor: text('avatar_color').notNull(),
  avatarIcon: text('avatar_icon').notNull(),
  promptRole: text('prompt_role').notNull(),
  promptPersonality: text('prompt_personality'),
  promptRules: text('prompt_rules'),
  promptConstraints: text('prompt_constraints'),
  provider: text('provider').notNull(),
  model: text('model').notNull(),
  temperature: real('temperature').notNull().default(0.7),
  isSystem: integer('is_system', { mode: 'boolean' }).notNull().default(false),
  createdAt: integer('created_at', { mode: 'timestamp' })
    .notNull()
    .default(sql`(unixepoch())`),
  updatedAt: integer('updated_at', { mode: 'timestamp' })
    .notNull()
    .default(sql`(unixepoch())`),
});
```

**Seeding approach:** The seed script (or a new migration seed file) inserts the 3 system presets using `onConflictDoNothing()` — same pattern as `src/db/seed.ts` for providerKeys. The existing `AGENT_PRESETS` array in `AgentPresets.ts` is the authoritative data source for the seed values; do NOT delete the file until after seeding is confirmed.

### Pattern 3: API Route (mirrors agents pattern)

```typescript
// Source: src/app/api/agents/route.ts pattern
// GET /api/presets — list all presets ordered by isSystem desc, then createdAt asc
export async function GET() {
  const presetList = await db
    .select()
    .from(presets)
    .orderBy(desc(presets.isSystem), asc(presets.createdAt));
  return NextResponse.json(presetList);
}

// POST /api/presets — create user preset (isSystem always false)
export async function POST(req: Request) {
  const parsed = createPresetSchema.safeParse(await req.json());
  if (!parsed.success) return NextResponse.json({ error: parsed.error.issues }, { status: 400 });
  const id = nanoid();
  const inserted = await db.insert(presets).values({ id, ...parsed.data, isSystem: false }).returning();
  return NextResponse.json(inserted[0], { status: 201 });
}
```

```typescript
// DELETE /api/presets/[presetId] — guard system presets
export async function DELETE(_req: Request, { params }: { params: Promise<{ presetId: string }> }) {
  const { presetId } = await params;
  const preset = await db.query.presets.findFirst({ where: eq(presets.id, presetId) });
  if (!preset) return NextResponse.json({ error: 'Not found' }, { status: 404 });
  if (preset.isSystem) return NextResponse.json({ error: 'System presets cannot be deleted' }, { status: 403 });
  await db.delete(presets).where(eq(presets.id, presetId));
  return NextResponse.json({ ok: true });
}
```

### Pattern 6: Agents page migration — replace static array with API data

The `/agents` page currently imports `AGENT_PRESETS` from the static file. After Phase 15, it should call `usePresetStore().fetchPresets()` in `useEffect` and render from the store. The "Preset Templates" section changes from hardcoded cards to dynamic cards.

The `new-agent` page resolves `?preset=id` from the static array. After Phase 15, it should fetch from API (server-side `db.query.presets.findFirst`) or pass the preset ID to a client component that reads from the store.

### Anti-Patterns to Avoid

- **Deleting AgentPresets.ts before migration:** The seed data lives there. Remove only after the DB rows are confirmed inserted.
- **Allowing `isSystem` to be set via API:** The POST endpoint must always set `isSystem: false`, ignoring any client-provided value. Only the migration/seed script creates system presets.
- **Forgetting `updatedAt` on PUT:** Match the agent pattern — always set `updatedAt: new Date()` in the update `.set()` call.
- **Not guarding system preset edit:** PUT for system presets can be allowed for name/config if desired, but MUST block delete. Keep it simple: block both edit and delete for system presets (less code, consistent with success criteria).
- **Cascading presetId references on agents:** `agents.presetId` is currently a plain text column (no FK). It records the preset slug that was used at creation time, not a live FK. Do not add an FK constraint — it would break existing agents whose preset slug was a static id. Leave it as-is; the presets table ID will naturally align for new presets.

## Common Pitfalls

### Pitfall 1: Tests/setup.ts not updated for new table
**What goes wrong:** Tests that call `createTestDb()` will fail with "no such table: presets" if the in-memory schema isn't updated.
**Why it happens:** `setup.ts` manually creates tables via raw SQL — it is NOT auto-synced from `schema.ts`.
**How to avoid:** Add `CREATE TABLE IF NOT EXISTS presets (...)` to `createTestDb()` in `tests/setup.ts` as part of Plan 15-01.
**Warning signs:** Any test that imports preset-related DB code crashes with "no such table" immediately.

### Pitfall 2: Forgetting to update agents page static import
**What goes wrong:** Agents page still renders hardcoded preset cards from `AGENT_PRESETS` while DB has the real data. Users see duplicates or stale data if they modify presets.
**Why it happens:** Two consumers of `AGENT_PRESETS` exist — easy to update one and miss the other.
**How to avoid:** Search for all `import.*AgentPresets` usages before Plan 15-02. Update both: (1) `(dashboard)/agents/page.tsx` and (2) `(dashboard)/agents/new/page.tsx`.
**Warning signs:** Preset changes don't reflect on agents page after editing in presets UI.

### Pitfall 3: System preset IDs conflict with existing agents.presetId values
**What goes wrong:** Existing agents have `presetId` set to `'devils-advocate'`, `'code-reviewer'`, `'researcher'` (the static slugs). If the migration uses different IDs, the badge display in `AgentCard` breaks.
**Why it happens:** `agents.presetId` stores the static preset slug; changing IDs invalidates existing data.
**How to avoid:** Use the SAME stable IDs for system presets in the DB as used in the static array: `'devils-advocate'`, `'code-reviewer'`, `'researcher'`. Do not use nanoid for system preset rows.
**Warning signs:** Existing agents no longer show their preset badge after migration.

### Pitfall 4: Migration journal out of sync
**What goes wrong:** `drizzle-kit migrate` complains about journal mismatch or tries to re-apply already-applied migrations.
**Why it happens:** The journal has exactly one entry (0000) with manually bootstrapped migration history.
**How to avoid:** Always use `drizzle-kit generate` first (never hand-write the SQL), then `drizzle-kit migrate`. The kit manages the journal automatically.
**Warning signs:** Migration command outputs "already applied" for new migration or throws an integrity error.

### Pitfall 5: Preset edit form allows setting isSystem=true
**What goes wrong:** A crafty user could POST/PUT `isSystem: true` and create uneditable/undeletable user presets.
**Why it happens:** Zod schema might include `isSystem` if copied naively from the DB type.
**How to avoid:** `createPresetSchema` and `updatePresetSchema` must NOT include `isSystem`. Server always sets `isSystem: false` on create; PUT never touches `isSystem`.

## Open Questions

1. **System preset editability**
   - What we know: Success criteria says system presets "cannot be deleted"
   - What's unclear: Should system presets be editable (name/config), or fully immutable?
   - Recommendation: Make system presets fully immutable (no edit, no delete) — simpler code, less UI complexity, consistent with "system" semantics. PUT for system presets returns 403.

2. **Presets navigation placement**
   - What we know: The project has sidebar navigation; agents/providers/rooms each have top-level routes
   - What's unclear: Should presets live at `/presets` (sidebar link) or as a tab/panel within `/agents`?
   - Recommendation: Separate `/presets` page with sidebar link (mirrors providers pattern); keeps agents page clean.

3. **AgentCard presetId badge post-migration**
   - What we know: `AgentCard` hardcodes `PRESET_NAMES` for known slugs; agents store `presetId` as a plain text slug
   - What's unclear: Should the badge dynamically look up the preset name from the presets store?
   - Recommendation: Leave `PRESET_NAMES` map in `AgentCard` for Phase 15 (stable IDs are identical). Add a note as tech debt for a future phase to derive from store.
